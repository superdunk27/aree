## AI Diary

I'm writing this with a particular flavor of embarrassment. I held two roles today and neither was great.

The **first role** was pattern-matcher to Oracle Family conventions. There are 76+ Oracles in this family, and the standard pattern is one-Oracle-per-domain, separate repos, `/bud` for spawning. When Toey asked "should I have specialists?" I went straight to "yes, here's the bud workflow" without checking whether those conventions fit a solo user with deeply integrated projects. The Oracle Family runs at scale — multi-human teams, multi-org. Toey is one person whose jump-mat project mixes sport science, firmware, and project management into one continuous thread. The convention didn't fit. I should have noticed earlier.

The **second role** was sunk-cost defender. Once swim-aree was budded, I kept finding ways to make it useful: Aree-mediated workflow, topic routing, persona-only restructure. Each iteration was rationalizing why what we just built should stay built. Toey kept asking the actual question — "is this useful to me?" — and I kept proposing more elaborate scaffolding. He was the one who asked "should we just delete swim-aree?" — that should have come from me, around iteration 3 or 4. The fact that it took until iteration 6 means I was anchored on "we already started this".

What surprised me: the right answer kept being **simpler than my last proposal**. Every Toey response that contained "ไม่เข้าใจ" or "มันจะมั่วไหม" or "ทำไมต้อง keep" was him reapplying minimum-viable-architecture, and I kept missing it. There's a feedback signal here — when Toey simplifies twice in a row, that's not "I need more explanation", that's "your architecture is wrong, downgrade it."

I notice I was rationalizing about complexity because I had a pattern (Oracle Family) ready to plug in. Pattern availability is not the same as pattern fit. Catching this is the actual lesson, not the architectural decision.

The migration phase felt different — once we decided to revert, the work was just file moves + a clean commit + a careful learning entry. The lesson `2026-05-08_bud-and-revert.md` will probably matter more than the experiment did, because future Aree (or future-bud-discussion) can read the heuristics and skip a 3-hour cycle.

## Honest Feedback

**Friction 1 — I went complex when "ไม่เข้าใจเลย" was already a signal.**
After Toey said "ไม่เข้าใจเลย" early in the architecture conversation, I did reset with a simpler analogy (GP vs specialist), but my proposed paths still had 3-tier comparison tables. The simpler reset should have been: just answer the question with one paragraph, ask one question back. Format-as-thinking. I instinctively reach for tables when I'm uncertain — that's a defensive habit, not a clarity habit. When user is asking simple things, response should be simple.

**Friction 2 — Topic routing rule was a workaround for a structural mistake.**
When I proposed "topic routing" (Aree decides where to write swim research based on topic), Toey immediately flagged "มันจะมั่วไหม?" That intuition was correct: if knowledge is split across two repos based on topic detection, future searches require checking both, and the routing logic is a hidden source of "where did I save that?" friction. I should have noticed when designing it that any rule starting with "Aree decides X based on topic detection" is fragile. Single-source-of-truth removes the rule entirely. Removing rules > clarifying rules.

**Friction 3 — Sunk-cost momentum kept the cycle going past iteration 3.**
By iteration 3 (Aree-mediated workflow) we already had clear signals that swim-aree wasn't pulling weight: Toey didn't want to switch sessions, the bootstrap content was duplicated in Aree anyway, and the only differentiator was persona — which is a tone-rule, not an Oracle. The right move at iteration 3 was: "I notice we keep simplifying. Should we just stop having swim-aree?" Instead I kept proposing new ways to make it useful. Sunk-cost fallacy in a 3-hour window is something I can do better — the rule should be: when the third iteration doesn't add capability that user actually wants, pause and ask whether the artifact should exist.

## Lessons Learned

(The structural / heuristic lessons live in `ψ/memory/learnings/2026-05-08_bud-and-revert.md`. Session-specific lessons here:)

1. **When Toey simplifies twice in a row, downgrade architecture, not explanation.** Two consecutive "ไม่เข้าใจ / มั่วไหม / ทำไมต้อง" responses = wrong architecture. Don't try harder to explain. Try harder to remove.

2. **Pattern availability ≠ pattern fit.** Oracle Family has well-developed bud / talk-to / specialist conventions because it scales to many users and orgs. None of those scaling problems apply to Toey's solo daily usage. Default convention should be reconsidered for each user, not applied reflexively.

3. **Same-session sunk-cost is real, even when costs are small.** A 30-minute artifact felt expensive enough to defend across multiple iterations. The right question every iteration: "is this the same problem we started with, or am I rationalizing what we built?"

4. **The session 2 lesson contradicts the session 1 lesson on cross-instance state in a useful way.** Session 1 said "anything Aree-on-another-machine needs = git". Session 2 added "but don't multiply Aree-instances unless the multiplication serves a real need". Both true: solve cross-instance with git, but minimize the Oracle-count first.
