## Lessons Learned

Writing the lesson file in parallel: `ψ/memory/learnings/2026-05-14_verify-output-not-just-fix.md` — when fixing a system, the verification has to test the *work output*, not just *the absence of the broken symptom*. The trade-bot today had three symptoms removed (DRY_RUN mismatch, ValueError loop, no watchdog) but the work output (`signal.bar`) was still missing because of bug #4. Until bug #4 was found and fixed, all the other fixes were technically correct but functionally incomplete. The discipline: define "fixed" in terms of the system producing its intended output, then verify *that* before declaring done.

This generalizes the morning's "runtime not docs" lesson (Toey asked "how do I send a pic" → make `pic` the runtime command) and yesterday's "ask where before what" lesson (Toey's bug report → ask which device first) — they all share the structure of "match the diagnostic level to the question the user actually has." For trade-bot the user's question wasn't "is the process restartable" or "is the supertrend bug fixed" — it was "is the bot doing what I built it to do."
