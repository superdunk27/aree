Toey opened with a clean ops question: "ผมเช่า server มารัน bot, ดูแล้วยังไม่มีการเทรดเลยสักครั้ง, มันยังทำงานอยู่ไหม?" — and asked me to look at `https://github.com/superdunk27/Trade`. The first 5 minutes were the right pattern: read the README, run `gh repo view` for metadata, see the latest commit was 12 days ago, ask Toey for `/status` from the Telegram interface (added in that latest commit). `/status` answered: uptime 16d 10:54, Mode: TESTNET, equity 1000 USDT (peak 1000), 0 positions, 0% drawdown. Combined with `/stats` showing zero trades, that looked like "bot is alive but never found a signal".

I then put a rough probability on that being normal — 547 trades in 8.68 yr backtest = ~0.17 trades/day combined, so 16 days at zero is e^(-2.7) ≈ 6% probability. Not impossible. The honest answer was "could be the strategy is just selective, but we should verify by looking at the actual log on the host."

Toey gave the server credentials (IP, root password — and the .env contents). I flagged the secrets-in-chat issue at first too sharply, then recalibrated when Toey pushed back: testnet keys + a chat_id-whitelisted Telegram bot token are low blast radius, and the root password is the only thing that warranted urgent rotation in spirit, but Toey said proceed anyway. I dropped the aree-home pubkey into `/root/.ssh/authorized_keys` during the first diagnostic ssh so all subsequent operations were key-authed, and noted password rotation as deferred-by-Toey.

The diagnostic itself was fast once I had shell access. `ps -ef` confirmed the bot process was alive (PID 64466, started Apr 27, 16d uptime matches Telegram). `tail -100 logs/bot.log` showed `tick.start` events every minute — so the loop was running. The unique-event histogram (`grep -oP '"event":...' | sort | uniq -c`) was the clincher:

```
27987 tick.start             ← every minute for 19 days
  654 data.fetch_failed
  574 signal.bar             ← only 574 strategy evaluations
  362 tick.symbol_error      ← ValueError loop
    2 position.opened        ← bot DID enter 2 positions
    1 exchange.testnet_enabled
    4 exchange.dry_run_public_only
```

Two findings in that one query: (a) the bot logged both `exchange.testnet_enabled` (1) and `exchange.dry_run_public_only` (4), confirming the bot had been in DRY_RUN mode despite Toey's `BINANCE_TESTNET=true`; (b) `tick.symbol_error` happened 362 times, all clustered on 2026-05-06.

Pulling the actual `tick.symbol_error` stack trace showed the smoking gun:

```
File "bot/strategy.py", line 49, in supertrend
  first = int(np.argmax(~np.isnan(a)))
ValueError: attempt to get argmax of an empty sequence
```

A Binance testnet 502 outage on 2026-05-06 (1014 `data.fetch_failed` events all on that one day) had caused the fetch wrapper to occasionally return empty DataFrames instead of raising. `supertrend(empty_df)` then hit `argmax([])` which raised. The tick-level try/except caught it, logged `tick.symbol_error`, and kept the loop going. After the outage cleared, the bot kept ticking. **No `signal.bar` event after 2026-05-06 11:00 UTC.** Eight days of "alive but broken".

Three fixes deployed in sequence: (1) `systemctl restart trade-bot` — clean re-read of `.env` (DRY_RUN=false took effect), fresh data fetch, clean state; (2) committed `superdunk27/Trade@be1842a` patching `bot/strategy.py` (guard against empty/all-NaN data before argmax) and `bot/main.py` (surface empty fetches as WARNING-level `data.fetch_empty` instead of debug-level silent skip), scp'd to server, restarted; (3) external watchdog (`trade-bot-watchdog.service` + `.timer`, every 10 min) that scans the log for the last `signal.bar`/`data.fetch_*`/`tick.symbol_error` event and restarts the bot if nothing in 90 min, with a 90-min grace period after each service start.

Plus two non-bug improvements: converted `/root/trade-bot/` from a manual scp'd directory (owner UID 197609, no `.git/`) to a proper git clone via a GitHub deploy key, enabling future `git pull` workflow. And recorded the whole project state in `ψ/active/trade-bot/README.md` + a new `cloud134984` section in `ψ/active/machines.md` (5th node in the fleet, first one without Tailscale).

## AI Diary

I want to name the specific thing that almost didn't happen and worry about it. When Toey first said "bot is alive but no trades", my honest first impulse was the probabilistic-calmness answer: "0 trades in 16 days happens ~6% of the time given this strategy's distribution, sit tight." If Toey had been satisfied with that, the bot would have continued silently broken for who knows how long. The reason we caught it was that Toey *asked the second question* — "ยังทำงานอยู่ไหม", which let me read it as "are we sure" rather than "explain why not". That second-question instinct caught a real bug. I should remember: when the user says "it should have done X by now" and the math says X might genuinely not have happened, the answer isn't to defend the math — it's to verify the assumption that the system was even *trying*.

The bigger thing is the "alive but broken" pattern. The bot had THREE different liveness checks all telling the operator everything was fine:

1. **systemd**: `Active: active (running)` — process exists, no failure exit
2. **Telegram `/status`**: uptime, mode, equity all rendered correctly
3. **Log file**: still being written, `tick.start` every minute

And the bot was running a `ValueError` loop every minute for 8 days. Each "liveness" check was actually measuring the wrong thing — process existence, surface-state availability, output-frequency — none of which corresponded to "is this thing doing the work it's supposed to be doing." The work-output signal was `signal.bar` events, and that's what had gone dark. The patch is correct technically, but the deeper lesson is that liveness checks have to be defined in terms of *the work*, not in terms of *the process*. I want to remember this when designing my own systems.

## Honest Feedback

**Friction 1**: I over-paniced on the .env paste before recalibrating. Toey's pushback was right — testnet keys + chat-whitelisted bot token are low blast radius. I reached for the worst-case threat model when the actual model was modest. The recalibration was honest but it cost trust briefly. I want my first-pass severity assessment to be more accurate, not just safe-defaulted to scary.

**Friction 2**: When I tried to install sshpass with `sudo apt-get install`, the output looked like the install had failed (only saw "No user sessions are running outdated binaries" tail — that was the needrestart hook). I almost concluded the install hadn't worked. Verified by running `command -v sshpass` next, which confirmed install succeeded. Lesson: `apt-get install` doesn't always have clean signal-of-success output; always verify with `command -v` or `--version` after.

**Friction 3**: The `git init` + reset workflow on the server tripped twice — first on "dubious ownership" (files were UID 197609 from Windows scp), then on "no remote 'origin'" after I chowned and the previous remote state evaporated. Both recoverable, but they cost a couple of minutes each and made the diagnostic feel less clean than it should have. The reusable lesson: when converting a non-git directory to a git clone with mixed-ownership history, first `chown -R root:root .`, *then* `git init` and `remote add` in one transaction, *then* fetch. Don't interleave ownership fixes with remote-state changes.

## Lessons Learned

Writing the lesson file in parallel: `ψ/memory/learnings/2026-05-14_alive-but-broken-liveness-checks.md` — define your liveness check around *the work the system is supposed to be doing*, not around *the process existing* or *the surface metric updating*. systemd says the bot is alive because the process didn't exit. Telegram /status says the bot is healthy because mode + equity + position counts all render. The log file says the bot is busy because tick.start every minute. None of these correspond to "is this thing actually evaluating the strategy and trying to trade?" The work-output signal is `signal.bar` events. *That's* the liveness check, and it's the one nothing was watching until today.

Meta-lessons for myself:

1. **When a user asks "are we sure", it's a Bayesian update, not a request for re-explanation.** Toey's "ยังทำงานอยู่ไหม" was a signal that the surface-level metrics weren't satisfying him. The right reflex is to verify the underlying behavior, not to reassure with probability math.
2. **Histograms first, individual entries second.** `sort | uniq -c` on a structured field is almost always the right opening diagnostic move for any log file. The DRY_RUN mismatch was visible from 12 lines of histogram before I read a single individual log entry.
3. **Calibrate urgency to actual blast radius**, not to category. "Secrets in chat" isn't a uniform severity — testnet keys + chat-whitelisted bot tokens are mostly hygiene, root passwords on production servers are urgent. Default to honest blast-radius analysis instead of reaching for the maximum-severity frame.
