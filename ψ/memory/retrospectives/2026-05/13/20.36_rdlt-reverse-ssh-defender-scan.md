No malware, no failing hardware.

Toey then asked how to permanently disable Defender. ## Lessons Learned

Writing the lesson file in parallel: `ψ/memory/learnings/2026-05-13_openssh-on-windows-gotchas.md` covers the reusable bits: (a) Windows usernames don't follow Linux conventions, always probe with `Get-LocalUser`; (b) admin accounts use `administrators_authorized_keys` not `~/.ssh/authorized_keys`, with strict ACL that `icacls /inheritance:r /grant Administrators:F /grant SYSTEM:F` produces; (c) `OpenSSH/Operational` event log is the canonical place to debug Windows SSH auth failures — it tells you whether the user is `Invalid user` (wrong name) vs `Permission denied (publickey)` (right name, wrong key); (d) Tailscale SSH server on Windows is not currently a reliable substitute for OpenSSH; (e) `MsMpEng` high CPU after a Windows Update batch is the expected post-install verification scan, not an indication of malware or hardware failure.

Meta-lessons for myself, again:

1. **Three anchor failures in one day. The pattern is the lesson, not the individual instances.** I keep treating the most-available signal (recent commit / Linux convention / UI label) as the answer. The discipline isn't hard in the abstract; it has to become reflex *before* the cheaper validation question. Tomorrow: when proposing a name or value derived from any inference chain, write down the one-line probe that would prove or disprove it, and run that first.
